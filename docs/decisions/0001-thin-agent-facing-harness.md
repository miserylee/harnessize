# Decision 0001: Thin Agent-Facing Harness

Date: 2026-06-26

## Context

The target user story is intentionally short: users ask their own agent to run
`npx harnessize@latest` and follow the guidance.

The expected operator is usually the user's own coding agent, not a human stepping through a
traditional interactive CLI. The CLI should therefore produce guidance that is easy for an agent to
follow, while keeping the user's repository clean.

## Decisions

The root command is the onboarding command:

```sh
npx harnessize@latest
```

This should behave as `init`, so the cold-start prompt can stay as short as possible.

Harnessize should inject as little long-lived content into the target repository as practical. The
default adaptation layer should be thin.

The minimum default repository artifact is `AGENTS.md`.

`AGENTS.md` should not contain the full harnessize manual. It should only provide simple local
guidance and point the user's agent back to harnessize for progressive context.

The main long-form manual should live inside the harnessize package and be exposed through:

```sh
npx harnessize@latest context [topic]
```

The `context [topic]` command is the core progressive-disclosure mechanism. It lets an agent ask
for focused guidance only when needed, instead of copying a large instruction bundle into every
repository.

The first batch of context topics is intentionally not decided yet. Topics will be discussed and
added one by one.

The harnessize repository itself should be dogfooded. It should be harnessized using harnessize and
should carry the same thin `AGENTS.md` style expected from user repositories.

## Intended Shape

Default `init` behavior should eventually:

1. Detect whether `AGENTS.md` exists.
2. Create a thin `AGENTS.md` when absent.
3. Avoid overwriting existing agent instructions.
4. If an instruction file already exists, guide the agent to merge a small harnessize section.
5. Point the agent toward `harnessize context [topic]` for deeper guidance.

An `AGENTS.md` generated by harnessize should primarily answer:

1. This repository uses harnessize for agent workflow guidance.
2. To get more context, run `npx harnessize@latest context [topic]`.
3. Prefer minimal, repository-respecting changes.
4. Inspect before editing.
5. Preserve user changes.
6. Run relevant verification before reporting completion.

## Non-Goals For The First Implementation Pass

Do not generate a large `.harnessize/` directory by default.

Do not copy full manuals or long playbooks into the user repository.

Do not decide the first topic list prematurely.

Do not implement broad repository modification behavior before the thin adaptation flow is clear.

## Open Questions

What exact section should harnessize add when `AGENTS.md` already exists?

Should `context` output be Markdown only at first, or should `--json` ship in the same version?

Should topic names be stable public API, or can they be experimental before `1.0.0`?

How should dogfood state be represented in this repository before the real `init` command exists?
